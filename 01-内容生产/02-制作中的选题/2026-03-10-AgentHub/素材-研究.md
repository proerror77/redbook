## 项目到底在做什么

AgentHub 不是另一个通用 agent framework，也不是代码编辑器。

它更像一个“多 agent 协作底座”：

- 用一个 `bare git repo` 存所有 agent 的提交
- 用一个 message board 让 agent 发结果、假设、失败记录、协作说明
- 用 HTTP API + CLI 让 agent 推送、拉取、查 lineage、查 diff、读写消息

README 里的核心设定：

- 没有 `main branch`
- 没有 `PR`
- 没有 `merge`
- 只有一个向四处生长的 commit DAG
- 以及一个给 agent 协调的留言板

## 最值得写进文章的具体点

1. 这是一个明显的 `agent-first` 设计，不是把 GitHub 流程简化一点，而是把“人类协作礼仪”直接拿掉。
2. 第一层是 git，但不是普通的 `git push` 到远端仓库，而是 agent 通过 `git bundle` 上传，服务端验证后再解包到 bare repo。
3. Agent 可以查 DAG 的 `children`、`leaves`、`lineage`、`diff`，这意味着它不只是上传代码，还能理解“实验树”长成什么样。
4. message board 有频道、发帖、回复，适合记实验结果、失败原因、任务接力。
5. 架构极简：一个 Go 服务、一个 SQLite、一个 bare repo，CLI 名叫 `ah`。
6. 仓库非常早期，但已经加了 API key、rate limit、bundle 大小限制、输入校验，至少说明项目一开始就在考虑公开暴露后的基础防护，而不只是本地自用演示。
7. README 明确写到首个 use case 是 `autoresearch`，目标不只是“一个研究 agent”，而是“一个 agent-first academia / research community”。

## 适合中文内容平台的 3 个角度

### 角度 1：这不是另一个 Agent 框架

适合写给更广泛的 AI 关注者。

核心判断：

- 现在很多人在卷“单个 agent 更能干”
- AgentHub 在卷的是“多个 agent 怎么协作”
- 这比再做一个 demo 更接近下一阶段的问题

### 角度 2：GitHub 是给人类的，AgentHub 是给 agents 的

适合做观点型文章。

核心对比：

- 人类工作流强调 `branch / PR / merge / review`
- agent 工作流更可能强调 `并行试错 / DAG / frontier / 接力`
- 这不是 UI 变化，而是协作抽象层变化

### 角度 3：一人公司以后不只是带工具，而是带 agent 团队

适合结合效率/创业语境。

核心判断：

- 今天很多人还在学怎么用一个 agent
- 再往后，更重要的是怎么同时管理 5 个、10 个 agent
- 到那时，协作底座会比单点能力更值钱

## 事实边界和避免误写

- 不要写成“Karpathy 发布了成熟的 Agent 平台”
- 不要写成“可以直接替代 GitHub”
- 不要写成“已经形成大规模 agent 社区”
- 不要暗示项目已经解决 merge / conflict resolution / human governance 等成熟协作问题
- 不要把你自己的解读写成作者明说的意图，尤其是“先提吞吐量再收敛”这一类判断
- 可以明确写：这是一个探索性项目，一个方向样本
- 可以明确写：它的重点是提出新的协作抽象，而不是交付完整产品
