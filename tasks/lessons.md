### Lesson 001
- 日期：2026-03-03
- 场景：运行每日入口 `bash tools/daily.sh`（全量）并自动追加选题到 `01-内容生产/选题管理/00-选题记录.md`
- 问题：
  1) 终端/日志长时间无输出，看起来像卡住
  2) 选题池被追加了无意义的统计字段（如 Deck 列数/推文数），且同日多次运行会重复追加
- 根因：
  1) `python ... | tee` 使 stdout 变为 pipe，Python 默认 block-buffer
  2) 选题抽取用全局 `**...**` 正则，覆盖了报告里的加粗统计标题；且缺少同日去重/插入逻辑
- 修正动作：
  1) `run_daily.sh` 改为 `PYTHONUNBUFFERED=1 python3 -u ... | tee`，保证实时输出
  2) `append_topics_to_record()` 只抽取“编号列表中的加粗 topic”，并在同日 section 内去重插入
- 预防规则（Rule）：
  1) 任何 `python ... | tee` 一律加 `-u` 或 `PYTHONUNBUFFERED=1`
  2) 从 Markdown 抽取结构化信息必须用结构化锚点/模式，禁止全局扫面式 regex
  3) 每日/定时任务必须幂等：同日重复运行不应污染数据
- 下次触发信号：脚本运行 30s+ 无输出；选题记录出现“列数/推文数/统计”字样；同一日期重复出现多段 “X 每日研究发现”
- 验证结果：`bash tools/daily.sh` 输出连续进度；2026-03-03 选题追加为 AI/code/openai/agent/rust，重复运行不会再重复追加

### Lesson 002
- 日期：2026-03-06
- 场景：通过 `baoyu-post-to-x` 在 `https://x.com/compose/post` 自动发布普通帖子
- 问题：
  1) 编辑器里已经出现文本，但 `发帖` 按钮仍然 disabled
  2) 误把“DOM 里有字”当成“X 已认可该内容可提交”
- 根因：
  1) `x-browser.ts` 与 `x-video.ts` 使用 `document.execCommand('insertText')`，只改了编辑器 DOM，没有稳定触发 X composer 的内部状态更新
  2) 没有用真实输入链路或 CDP 原生输入验证按钮状态
- 修正动作：
  1) 将 regular post / video post 的文本输入改为 `Input.insertText`
  2) 在真实登录会话中回归验证 `tweetButton` 会从 disabled 变为 enabled
- 预防规则（Rule）：
  1) 对富文本编辑器，禁止把 DOM 注入当作输入完成；优先使用真实键盘/粘贴事件或 CDP 原生 `Input.insertText`
  2) 自动化发布前必须验证提交按钮状态，不能只检查编辑器中是否出现文本
  3) 当站点存在多个发布流（普通帖 / Article）时，先确认当前流程对应的编辑器与校验逻辑，再决定自动化策略
- 下次触发信号：编辑器有字但按钮灰掉；脚本日志显示“Typing text...”后无法提交；手动粘贴可发而自动化失败
- 验证结果：2026-03-06 在当前登录 X 会话中注入 110 字测试文本后，`tweetButton` 为 enabled，`buttonText` 为“发帖”

### Lesson 003
- 日期：2026-03-06
- 场景：将热点母稿改写并发布到 X
- 问题：
  1) 文案信息完整，但仍有“AI 写的”痕迹
  2) 发布前没有单独做平台化去 AI 味处理
- 根因：

  2) 本轮只用了 `baoyu-post-to-x` 做发布，没有先用专门的 X 写作改写流程把语言打磨成更像真人发言
- 修正动作：
  1) 以后发布 X 主帖前，先做一次“去 AI 味”改写：删解释腔、删对称排比、删过满铺陈，保留更强观点和更像口语的断句
  2) 将“是否像真人会直接发出来的话”加入发布前检查项
- 预防规则（Rule）：
  1) `baoyu-post-to-x` 只负责发布，不负责去 AI 味；创作与发布必须分开检查
  2) 从母稿改写到 X 时，优先用更短句、更强立场、更少“完整解释”，不要让文案像条理化说明书
  3) 发布前至少自检 3 件事：像不像观点表达、像不像本人会说的话、有没有“为了完整而完整”的痕迹
- 下次触发信号：文案过于工整、段落结构太齐、像在“解释一件事”而不是“说一句判断”
- 验证结果：待下次发 X 内容时执行该检查并复盘

### Lesson 004
- 日期：2026-03-07
- 场景：自动发布 X 单条长帖
- 问题：
  1) 纯自动化路径如果“填入即提交”，行为节奏过于机械
  2) 用户明确要求避免瞬间提交，以降低机器人感
- 根因：
  1) 之前的关注点放在“能不能发出去”，没有把发布节奏当成单独约束
  2) 自动化脚本默认在输入完成后立即点击提交
- 修正动作：
  1) 本次发布在文本填入后显式等待 7 秒，再点击 `发帖`
  2) 将“输入后停几秒再提交”纳入 X 发布默认检查项
- 预防规则（Rule）：
  1) 自动发布 X 时，除非用户另有要求，文本填入后至少停顿数秒再提交
  2) 发布流程不只校验技术成功，也要校验行为节奏是否过于像脚本
- 下次触发信号：发布脚本在 `Input.insertText` 后立刻点击提交；用户提到“不要像机器人”
- 验证结果：2026-03-07 发帖前等待 7 秒后成功发布，链接为 `https://x.com/0xcybersmile/status/2030110526861447639`

### Lesson 007
- 日期：2026-03-09
- 场景：撰写 RTK 工具推荐帖，用户指出语气“不够好”，并提供 `https://x.com/chenchengpro/status/2030616953983828456` 作为更生活化的参考
- 问题：
  1) 文案事实准确，但像“工具说明”而不像“人会顺手发出的推荐”
  2) 开头从工具定义切入，缺少真实使用场景、痛感和口语化入口
- 根因：
  1) 写法偏“先解释产品是什么”，而不是“先说用户为什么会在意”
  2) 过度依赖抽象判断句，例如“它不去重新发明 AI IDE”，工程味有了，人味不够
- 修正动作：
  1) 工具推荐类内容优先用“痛点/场景 -> 解决方式 -> 结果 -> 低门槛动作”四段式
  2) 默认先写得像开发者聊天，再补技术信息，而不是先写成产品简介
- 预防规则（Rule）：
  1) 写工具推荐帖时，第一句优先写用户场景或代价，禁止上来就说“最近看到一个很实用的工具”
  2) 机制解释最多一小句，重点放在“为什么值得试”和“试起来有多轻”
  3) 指标只保留最有感知的一组，不要把文案写成 README 摘要
  4) 如果用户给了参考文章，先抽它的语气骨架，再写当前内容，不能只复述事实
- 下次触发信号：开头出现“最近看到一个工具/项目”；正文连续两句都在解释原理；读起来像产品介绍而不是朋友推荐
- 验证结果：待下次写工具推荐类 X 帖或短文时，按该规则重写并复盘
