# Rust 系统软件安全边界

> 来源：https://corrode.dev/blog/bugs-rust-wont-catch/ | 最后更新：2026-04-29

## 核心内容

Corrode 这篇文章用 Canonical 对 Rust 重写版 GNU coreutils（uutils）的 CVE 审计做切口，讲一个适合技术内容表达的反直觉判断：

Rust 能显著减少内存安全问题，但它不会自动消除系统软件里的所有安全 bug。剩下的问题更靠近操作系统边界：路径、权限、syscall 时序、Unix 字节流、兼容性语义和错误传播。

可复用主论点：

- Rust 不是让开发者不用懂系统，而是把内存安全这类低层问题压下去，让开发者必须面对更真实的系统复杂性。
- 安全不是语言特性单独完成的，而是语言、运行时边界、操作系统语义和工程习惯共同形成的结果。
- 对系统工具来说，兼容性不是“体验细节”，而可能是安全边界的一部分。

## 可展开角度

### 角度一：Rust 安全神话的下半场

Hook：
「Rust 解决了 C/C++ 最痛的一类 bug，但这不等于系统软件从此安全。」

展开：
- Rust 可以管内存生命周期和数据竞争。
- 但它不理解 Unix 路径是不是被换成 symlink。
- 它不判断两次 syscall 之间文件系统状态是否被攻击者改变。
- 它也不会知道某个 CLI 与 GNU coreutils 的行为差异会不会让脚本误伤生产环境。

适合形态：
- X 长帖
- 面向工程管理者的短评
- “AI 写代码越来越强，但系统知识仍然重要”的类比文章

### 角度二：路径不是对象，是名字

Hook：
「系统编程里最容易误判的一点：路径不是文件本身，只是指向文件的一串名字。」

展开：
- TOCTOU 的核心是：第一次检查和第二次使用之间，路径可能已经指向了另一个对象。
- 安全代码不能长期信任 `Path` 字符串。
- 更可靠的做法是把操作锚定到 file descriptor。

适合形态：
- 技术科普短帖
- “为什么工程细节比语言选择更重要”的案例

### 角度三：兼容性是安全特性

Hook：
「重写经典工具时，bug-for-bug compatibility 有时不是包袱，而是安全要求。」

展开：
- shell 脚本、cron、CI、系统自动化依赖工具的既有语义。
- 如果 Rust 重写版行为和 GNU coreutils 不一致，可能不是“小差异”，而是生产事故入口。
- 对基础设施来说，用户依赖的是历史契约，不只是源码质量。

适合形态：
- X 观点帖
- 工程文化/基础设施维护文章

## 关键事实点

- 文章讨论的是 Rust 重写版 GNU coreutils：`uutils`。
- Canonical 审计并披露了一批与 uutils 相关的 CVE。
- 文章强调这些问题不是典型内存安全漏洞，而是系统边界、错误处理、权限和兼容性问题。
- 典型风险包括：TOCTOU、创建后再改权限、路径字符串误判、错误吞噬、panic 导致 DoS、Unix bytes 被强行转 UTF-8。

## 可复用表达

- Rust 不是安全终点，而是安全起点。
- 编译器能保证内存生命周期，但不能替你理解 Unix。
- 路径不是文件，字符串相等也不等于文件身份相等。
- 对基础工具来说，兼容性不是情怀，是生产契约。
