# AI 已经从模型战争，进入部署战争

如果你最近高频用 AI，大概率已经开始碰到一种很具体的不爽。

不是它不够聪明。

而是：

- 它太啰嗦
- 它太贵
- 它有些活根本不值得上云
- 它明明离你很近，却还要绕远路去服务器转一圈

说白了，AI 现在最烦人的问题，越来越不是“不会做”。

而是：

**做是能做，但做得不划算。**

这就是我最近越来越确定的一件事：

**AI 已经从模型战争，进入部署战争。**

过去大家比的是谁更强。

现在开始比的是另一套东西：

- 谁更省
- 谁更快
- 谁更稳
- 谁更靠近用户
- 谁更会把任务放在对的位置上

这看起来像工程问题。

但它很快会变成产品问题、成本问题、体验问题，最后甚至会变成商业模式问题。

因为未来真正拉开差距的，不一定是谁家模型多聪明。

而是谁先想明白：

**什么活该在云上做，什么活该在本地做，什么活应该直接在设备上做。**

## 一个很小的现象，已经把方向暴露出来了

最近有个项目在 Hacker News 上很热，叫 `Caveman`。

它干的事特别不高级。

不是发论文。
不是训新模型。
不是搞更复杂的 agent 框架。

它只是干了一件很土的事：

**让 Claude Code / Codex 少说废话。**

项目 README 给的数据很扎眼：

- 平均从 `1214` 个 token 降到 `294` 个 token
- 平均节省 `65%`
- headline 直接写：大概能砍掉 `75%` 的 token

更妙的是，它还专门强调：

**影响的是 output tokens，不是 reasoning tokens。**

翻成人话就是：

不是脑子变小了。
是嘴变小了。

这件事为什么重要？

因为它暴露了一个变化。

以前大家对 AI 的期待是：

“你先证明你很强。”

所以哪怕它啰嗦一点、慢一点、贵一点，用户也愿意忍。

但当 AI 真开始进入工作流，评价标准就变了。

你不再只关心它会不会。

你开始关心：

- 它每次到底要吃掉多少 token
- 它在多轮 agent workflow 里会不会把上下文拖爆
- 它解释这一大段，到底有没有必要
- 它这次帮到我的价值，配不配得上这次账单

这不是文风问题。

这是生产问题。

所以 `低 token` 这条线值得重视，不是因为它有趣。

而是因为它说明：

**AI 已经从“能不能做”，进入“值不值得这么做”的阶段了。**

## 下一步自然就来了：能不能别什么都走云

当你开始嫌 AI 贵、嫌 AI 慢、嫌 AI 太能说，下一步几乎是必然的：

你会开始问：

**这件事能不能干脆别上云？**

这就是为什么最近“本地 AI”突然又热了。

但很多人一说本地 AI，就容易讲偏。

本地 AI 真正有价值的地方，不是情怀。
也不是“我就是不想被大厂绑定”。

它真正解决的是三件现实问题：

1. 更低的边际成本
2. 更可控的延迟
3. 更强的数据控制权

以前本地模型最大的尴尬是：

能玩。
但很难接生产。

你可以本地聊两句。
但很难把它塞进 IDE、agent、脚本、CI、自动化流程里。

说白了，它更像一个玩具，不像一个基础设施。

最近这个局面开始松动了。

## 本地 AI 真正的转折点，不是模型变强一点

而是它开始长接口了。

### Ollama 这条线已经很明显

Ollama 官方文档已经把 `Claude Code` integration 正式写进去了。

重点不是“又能接一个工具”。

重点是它给出的姿态已经变了。

它不是把本地模型当聊天替代品。

它是在说：

open models 可以通过 `Anthropic-compatible API` 接进 Claude Code 这种真正有工作流深度的 coding agent。

这句话背后的意味是：

**本地模型开始能接正经活了。**

当然，现实约束也没有消失。

Ollama 文档也写得很老实：

- Claude Code 这类工具需要大 context window
- 推荐至少 `64k tokens`
- 像 `qwen3-coder` 这种量级，至少要 `24GB VRAM`

所以本地 AI 不是白嫖。

它只是把原来付给 API 的钱，换成了：

- 显卡
- 内存
- 机器
- 功耗
- 维护复杂度

但这恰恰说明它更成熟了。

因为一个东西真正进入生产，不是“完全没成本”。

而是：

**成本终于可以被认真计算了。**

### LM Studio 更值得看

如果说 Ollama 的意义是“本地模型开始能接 Claude Code”，
那 LM Studio 的意义更大：

**本地模型开始像后端服务了。**

LM Studio 官方开发文档已经明确写了：

- 本地 `REST API`
- `OpenAI-compatible` endpoint
- `Anthropic-compatible` endpoint
- `tool calling`
- `local agents with MCP`
- 可以跑在 server / CI / cloud instance 上的 headless daemon

这说明什么？

说明本地模型不再只能待在一个本地 GUI 聊天框里。

它开始能被：

- IDE 调用
- agent 调用
- 脚本调用
- CI 调用
- MCP 工具链调用

这就是一个分水岭。

以前本地模型像个爱好者设备。
现在它开始像基础设施。

一旦这一层成熟，你做 AI 产品时的默认思路就会被改写。

以后不是所有任务都默认上云了。

你会越来越频繁地问：

- 这个任务是不是高频到值得本地化
- 这个任务是不是敏感到必须本地化
- 这个任务是不是便宜活，根本不该占用云端高价推理

所以本地 AI 的真正价值，不是“离线聊天”。

而是：

**把一部分本来不该上云的活，拿回来。**

## 然后会发生第三步：agent 开始往设备上长

如果说本地 AI 是“把推理拉回自己的机器”，
那端侧模型更进一步。

它问的是：

**能不能把 agent 直接放到设备上。**

很多人以前一讲 on-device AI，就会自动脑补成：

- 小号 ChatGPT
- 离线玩具
- 厂商秀肌肉的 demo

但 Google 这次关于 `Gemma 4` 的一整套动作，我觉得已经不能再这么看了。

## Gemma 4 真正释放的信号，不是“又一个模型”

Google 官方博客的标题非常直接：

`Bring state-of-the-art agentic skills to the edge with Gemma 4`

不是把聊天带到手机。
不是把推理加速做快一点。

而是：

**把 state-of-the-art 的 agentic skills 带到 edge。**

这已经不是在讲“本地聊天模型”了。

它讲的是：

- `multi-step planning`
- `autonomous action`
- `offline code generation`
- `agents and autonomous AI use cases running directly on-device`

你把这些词摆在一起看，就会发现一件事：

Google 这次押注的，不是“手机也能跑模型”。

而是：

**手机、终端、边缘设备上，会开始出现 agent runtime。**

Gemma 4 model card 给出的信号也很强：

- 最长 `256K` context
- 从 `E2B` 到 `31B` 的完整尺寸带
- 小模型明确写 `optimized for on-device`
- 支持 `native function-calling`
- 明确强调 `coding & agentic capabilities`
- 甚至明确提到 `high-end phones`

这不是一个厂商写给开发者的空话。

这是在认真铺一条部署路线：

从云，到本地，到 edge，开始出现完整链路了。

## 最值得重视的不是 Gemma 4，而是 AI Edge Gallery 里的那几个字

如果模型层还可以被看成“能力储备”，
那 `Google AI Edge Gallery` 就已经是产品样板了。

截至 `2026-04-06`，App Store 页面上最有意思的几个词是：

- `Gemma 4 is here`
- `Run the latest high-performance models fully offline`
- `Agent Skills`
- `Thinking Mode in AI Chat`
- `100% On-Device Privacy`

这几句放在一起，其实已经把未来一层产品形态讲出来了。

重点不是“手机也能离线聊天”。

重点是 `Agent Skills`。

Google 官方博客里对这东西的描述更直白：

- 用 skill 查 Wikipedia 做知识增强
- 把视频和文本转成 summary、flashcards、graphs
- 接其他模型做 TTS、图像生成、音乐生成
- 用一段对话串起一个完整 workflow

你看到这里，其实应该换个看法了。

AI Edge Gallery 不是在演示“本地跑模型”。

它是在演示：

**设备本身，开始变成 agent 的运行容器。**

而设备这层一旦开始有 agent，事情会很不一样。

因为设备天然离这些东西最近：

- 用户
- 传感器
- 语音
- 相机
- 位置
- 系统动作
- 个人隐私数据

这意味着未来很多体验，最优解可能根本不是“请求一个云端超级大脑”。

而是：

**先在设备上判断，先在设备上行动，只有复杂部分再上云。**

## 所以，真正的变化不是模型更小了

而是 AI 的优化顺序彻底变了。

以前大家先问：

- 谁最强
- 谁最准
- 谁更像人

现在真正做产品的人，会越来越先问：

- 这件事值不值得走云
- 这件事最贵的是 token 还是延迟
- 这件事是不是高频便宜活，应该本地做
- 这件事是不是隐私敏感，应该端侧做
- 这件事是不是一定要用最强模型，还是够用就行

这就是我说的“模型战争”变成“部署战争”。

因为下一轮真正拉开差距的，不是某个榜单上多了 2 分。

而是：

**谁先学会把不同任务，放到不同位置去跑。**

## 一个更现实的未来分层

我现在越来越相信，未来 AI 产品会慢慢分成三层。

### 云端

负责最复杂、最开放、最需要高能力的任务。

比如：

- 跨领域复杂研究
- 大规模内容生成
- 长链推理
- 跨系统编排

### 本地

负责高频、可控、想降本、想掌握数据的任务。

比如：

- 代码辅助
- 私有知识库问答
- 常见结构化处理
- 个人工作流自动化

### 端侧

负责低延迟、高隐私、强实时、依赖设备上下文的任务。

比如：

- 手机上的 personal agent
- 本地字幕、录音、摘要
- 相机、语音、位置相关动作
- 离线环境下的即时辅助

真正重要的是：

这三层不是互相替代。

它们更像一个分工系统。

谁先把这三层分对，谁的产品就会更强。

## 最后一句更直白的话

如果你今天还是只盯着：

- 跑分
- 上下文
- 价格
- 模型名字

那你看的是上一轮战争。

下一轮真正会决定输赢的，是另一件事：

**你到底有没有把对的活，放到对的位置上。**

`Caveman` 在砍 token，
`Claude Code + Ollama/LM Studio` 在把一部分活往本地拉，
`Gemma 4 + AI Edge Gallery` 在把 agent 往设备上推。

看起来像三条新闻。

其实它们在说同一件事：

AI 已经开始从“能力展示”，走向“部署优化”。

先压 token。
再拉本地。
最后推到端上。

这不是小修小补。

这是下一轮 AI 产品竞争真正会拉开差距的地方。
