# Redbook 系统优化方法参考

> 这些规则从主 playbook 移出，避免每次会话都加载长篇背景。需要做系统维护、记忆治理、存储治理或周复盘时再读取本文件。

## 使用原则

- 主流程只执行 `docs/shared/redbook-playbook.md` 的 lane 和硬门槛。
- 本文件是参考库，不是每个内容任务的 mandatory checklist。
- 当用户指出流程卡顿、上下文过载、任务状态混乱、存储路由错误时，再按相关章节取用。

## Token 与上下文

1. 懒加载：默认不读日记、本体画像、历史任务长文件；只在任务需要时读相关片段。
2. 模块化身份：涉及身份/价值观/关系/规划时，按需读取对应文件，不整包加载。
3. Skill 渐进加载：简单任务只读 skill 主文件，复杂任务再读 scripts、templates、references。
4. 指令遵循标志：Redbook 会话回复以 `✓` 开头；若做不到，优先判断上下文是否过载。
5. 会话边界：一个会话尽量聚焦一个问题，完成后归档到 `tasks/progress.md`。
6. 上下文阈值：发现重复读取大文件、规则互相打架、执行明显变慢时，优先瘦身当前上下文。

## 记忆与知识沉淀

7. 三层记忆：情景记忆保存原始经历，语义记忆沉淀方法，强制规则只保留必须遵守的约束。
8. 记忆提炼：重复模式达到 3 次或完成重大项目后，再抽象成可执行规则。
9. 日记合并：日记类材料按月合并，避免每日碎片长期污染检索。
10. Wiki 优先：内容素材、金句、案例、框架优先沉淀到 `wiki/素材/`，不要恢复旧 `02-内容素材库` 主链。

## 任务与状态

11. 时间意图捕获：用户说“明天/下周/记得/稍后 + 动作”时，记录到活跃任务或待办。
12. 发布状态更新：用户明确说“已经发布”时，归档内容并补发布记录。
13. 任务归档：活跃任务放 `tasks/active.md`，历史任务归档，`tasks/todo.md` 只作兼容历史账本。
14. Harness 分型：只有计划内容使用完整 `research -> draft -> review -> publish -> retrospect`；研究、wiki、工具修复可直接关闭。
15. 进度记录：非平凡任务结束前更新 `tasks/progress.md`，保留完成、遗留、下次优先项。

## 存储路由

16. 工作区边界：所有 Redbook 输出保存在 `/Users/proerror/Documents/redbook/` 内。
17. 智能存储：文章进 `01-内容生产/02-制作中的选题/`，已发布进 `03-已发布的选题/`，选题进 `选题管理/`，素材进 `wiki/素材/`。
18. 数据不物理删除：默认归档或 `git rm --cached`，不直接删除用户数据。
19. 每周存储复盘：检查错误存储、重复产物、生成缓存和可归档历史文件。

## 迭代与质量

20. 八步自我迭代：触发、复现、分析、设计、验证、实施、文档化、提交。
21. 对抗式纠错：给建议前主动检查漏洞和备选方案。
22. 主动自动化：重复操作达到 3 次时，建议脚本化或 lane 化。
23. 代码质量：遵循项目风格、错误处理、安全边界，避免为了流程优化引入新依赖。
24. 决策确认：删除、覆盖、外部发布等不可逆或高副作用动作必须有明确授权。
25. 周维护提醒：每周复盘数据、任务、存储、wiki 健康和重复规则。
