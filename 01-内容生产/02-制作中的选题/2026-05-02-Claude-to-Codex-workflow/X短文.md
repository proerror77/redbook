# X 短文：从 Everything Claude 到 Almost Everything Codex

我以前几乎是 Everything Claude。

现在已经接近 almost everything Codex。

过去 10 个月，Coding Agent 已经从“帮我写代码的工具”，变成了我最重要的工作伙伴之一。

这个变化不是因为我突然不喜欢 Claude 了。相反，Claude Code 很长时间都是我最重要的 agent 工作入口。

真正变的是：Codex 追上来的速度，以及它在产品端展现出来的完成度。

一开始，Agent 只是帮我完成一个具体任务：修 bug、写脚本、补文档。

后来我开始围绕 Agent 重新设计整套工作流。

量化交易、软件开发、社交媒体经营，这些赛道表面完全不同，但它们对 Coding Agent 的要求其实越来越像：

不是只会补一段代码，而是能理解项目状态，处理中途错误，调用工具，保留上下文，并把结果带到一个可验证的完成状态。

过去，这种能力更多依赖 Claude Code，加上我自己搭的 harness 和 workflow。

但这几个月 Codex 明显追上来了。

尤其是 Codex App 加上 GPT-5.5 的 long task 能力之后，我第一次很强烈地感觉到：

很多以前必须靠复杂外部工程系统才能跑完的事情，现在开始可以直接在 Codex App 里完成。

这也是我这几个月最大的感受：

6 个月前很先进的 workflow，6 个月后可能就不一定适用了。

工具在变，模型在变，Agent 的能力边界也在变。

所以真正重要的不是记住某一套固定工程方法，而是理解背后的原理：如何 harness agent。

也就是，不只是让 Agent 回答问题，而是让它在真实项目里持续推进、验证结果、修正错误，直到事情真的完成。

Codex 过去几个月的进步非常明显。

尤其是 @OpenAI 在工程系统、工具调用、项目上下文、长任务执行这些方向上的推进，作为重度用户是能明显感受到的。

从 Claude 到 Codex，对我来说不是一次简单的工具迁移。

更像是一次工作方式的迁移。

希望 Codex 继续进步。Coding Agent 这个方向，真的越来越有意思了。
