## 案例收集标准

好案例的特征：
1. **真实可信**：有具体数据和细节
2. **有代表性**：能说明普遍规律
3. **有启发性**：能给读者带来思考
4. **可验证**：有来源出处

- 案例标题 / 案例来源 / 案例背景 / 具体过程 / 结果数据 / 关键要素 / 可复用点 / 适用选题

## 案例：陈言 - 小红书 AI 账号

**背景**：2 年做到 10 万+粉丝，主要收入来自接广告

### 关键路径

1. **双边市场定位**：C 端（AI 工具测评需求）+ B 端（新产品曝光需求）同时设计
2. **MVP 验证**：同时启动汽车/AI/亲子三个账号，AI 数据最稳定，停掉另外两个
3. **竞争分析矩阵**：横轴（整活/讲技术/科普/讲工具）× 纵轴（体量/竞争力），找到空白地带"AI 场景实验室"
4. **封面优化**：用 AI 做封面后点击转化率从 5% 翻到 10%，稳定 8%-9%
5. **时机判断**：2023 年 12 月开始，踩在 ChatGPT 风口上

### 结果数据

- 粉丝：10 万+
- 笔记数：600 篇（340+ 篇纯 AI 产品介绍）
- 封面转化率：8%-9%（行业平均 5%）

### 可复用点

1. 定位阶段就要想清楚内容形式是否适合接广告
2. 冷启动时同时测试多个方向，让数据说话
3. 画竞争分析矩阵，找空白地带
4. 封面设计是转化率的杠杆点
5. 选题空间要大，至少能做 50 期内容

### 适用选题

- 小红书账号运营方法论 / AI 账号如何变现 / 内容创作的产品思维 / 一人公司如何做内容

## 案例：@karry_viber - OpenClaw 内容案例集

**来源**：X.com 推文（2026-02-15 收集，19 条，平均互动 86 赞）

### 核心案例

**旧手机改造 AI 助手**（435 赞）
- 痛点：豆包手机还在画饼，我已经实现了
- 对比：官方产品 vs 个人实现，门槛降低（旧手机 + 教程）

**OpenClaw 自主发推**（214 赞）
- 自动化演示：让产品自己发推证明能力
- 透明度：公开操作流程（打开 X → 找按钮 → 输入 → 点发送，全程无人触碰）

**拟人化叙事**（163 赞）
- 给 AI Agent 起名字（Orb）
- 技术隐喻：近视眼手术（视觉优化）、五感觉醒（功能模块化）、失忆（数据丢失）

### 写作手法提炼

1. **对比式开场**：「官方产品 X 还在做 Y，我已经用 Z 实现了」
2. **拟人化命名**：给技术产品起名字，用生物/人的特征描述
3. **流程透明化**：公开操作步骤，让读者可复现
4. **承认局限性**：不避讳问题，反而增加真实感

### 关键洞察

- 产品叙事的核心是"解决痛点"，而非"技术炫耀"
- 拟人化能大幅降低技术理解门槛
- 标签策略：产品标签 + 技术标签 + 行业标签 + 个人品牌

## 案例：Corrode - Rust 不会自动捕获的系统软件 bug

**来源**：Corrode blog，`Bugs Rust Won't Catch`

**背景**：文章围绕 Canonical 对 Rust 重写版 GNU coreutils（uutils）的 CVE 审计，说明真实 Rust 系统工具里仍会出现安全问题。

### 关键问题

1. **TOCTOU**：路径第一次检查和第二次使用之间可能被换成 symlink。
2. **权限窗口**：先创建目录或文件，再 `chmod`，中间会留下短暂风险窗口。
3. **路径身份误判**：路径字符串不同，不代表底层文件不同；字符串相等也不等于安全。
4. **Unix bytes 边界**：文件名、环境变量、命令输入不一定是 UTF-8。
5. **panic / unwrap 风险**：攻击者可控输入触发 panic 时，CLI 或系统任务可能变成 DoS。
6. **兼容性风险**：基础工具与 GNU coreutils 行为不一致，可能破坏脚本和生产自动化。

### 可复用点

- 用“Rust 不是银弹”切入，但不要停留在语言之争，要落到系统边界。
- 把“语言保证”和“操作系统语义”拆开讲，读者更容易形成判断。
- 兼容性可以被解释成安全特性，而不是保守主义。

### 适用选题

- Rust 系统软件安全边界 / AI 写代码为什么仍需要系统知识 / 基础设施重写的真实风险 / 语言安全与工程安全的区别
