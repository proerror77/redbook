# LiteLLM 这次事故最该让人警醒的，不是模型风险，是供应链风险

很多人做 AI 应用，最担心的是模型不稳定。

但这两天 LiteLLM 的事说明了一件更现实的事：

你最容易出事的地方，可能根本不是模型。

而是你顺手装进去、以为很安全的依赖。

2026 年 3 月 24 日，LiteLLM 社区和维护团队在 GitHub issue 里披露：

PyPI 上的 `litellm 1.82.7` 和 `1.82.8` 被投毒了。

更糟的是，`1.82.8` 里加进了一个 `litellm_init.pth` 文件。

如果你熟 Python，就知道这意味着什么。

这不是“你 import 之后才会执行”的问题。

而是 Python 解释器启动时，它就可能自动跑。

换句话说，危险不是“你开始用 LiteLLM 了”。

危险是“你装了它，这台机器就可能已经脏了”。

这次恶意代码做的事，也不是小打小闹。

公开分析里提到，它会收集环境变量、SSH key、云凭证、K8s 配置、Docker 配置、数据库密码、shell history，然后打包加密，发到攻击者控制的域名。

这件事为什么值得单独写？

因为 LiteLLM 不是一个冷门包。

它在很多 AI 工作流里，扮演的是“模型路由层”“统一接入层”“代理层”。

很多团队一上来就会装。

也正因为它位置太靠中间，一旦出事，伤害就不是某个 demo 挂了。

而是：

- 开发机可能泄密
- CI 可能泄密
- 容器镜像可能带毒
- 线上环境变量可能被带走

这才是 AI 应用时代最容易被低估的地方：

大家都在盯模型效果。

但真正能让团队直接进入事故状态的，往往是模型外面的那一层供应链。

我觉得这次 LiteLLM 事故，至少说明了 4 件事。

## 1. AI infra 包不是“普通依赖”

像 LiteLLM 这种包，天然就在流量入口、凭证入口、路由入口。

它们一旦出问题，波及面比普通业务依赖大得多。

所以以后装这类包，不能再按“pip install 一下再说”的心态来。

## 2. 版本号不是信任背书

维护团队后来在 timeline 里说得很清楚：

`1.82.7` 和 `1.82.8` 并不是通过官方 GitHub CI/CD 正常发出的版本，而是攻击者直接上传到了 PyPI。

这句话很关键。

它提醒你一件事：

**包管理器里出现一个新版本，不等于这个版本就可信。**

你得回到 GitHub release、CI 链路、签名和构建来源去交叉验证。

## 3. “只是本地开发机”这句话，以后不成立了

很多人会觉得，开发机中毒最多就是重装。

但 AI 开发环境和以前不一样。

现在大家本地机器上放的东西太多了：

- OpenAI key
- Claude key
- AWS 凭证
- GCP 凭证
- 各种数据库地址
- GitHub token
- 飞书、Slack、Webhook

如果这些都在环境变量里，那一次供应链投毒，就不再是“本地问题”。

它可能直接变成公司级安全事件。

## 4. AI 团队的默认安全动作，必须往前提

这次如果你真的装过受影响版本，我觉得正确反应不是“先等等看官方怎么说”。

而是默认按泄露处理：

1. 立刻定位是否安装过受影响版本
2. 检查 `site-packages` 里是否存在 `litellm_init.pth`
3. 轮换所有可能暴露的环境变量和配置凭证
4. 回看 CI、容器、构建机日志
5. 重新审视依赖 pin、镜像隔离和发布链路

这才叫工程反应。

不是转发一条安全新闻，就算处理过了。

## 我最后的判断

LiteLLM 这件事最值得记住的一句，不是“某个包被黑了”。

而是：

**AI 时代最脆弱的地方，往往不是模型本身，而是围着模型转的那圈供应链。**

以后谁还把 AI 工程理解成“换个更强模型 + 接个 API”。

谁就会在这种事情上吃大亏。

真正成熟的 AI 团队，不只会评估模型效果。

还会默认把：

- 依赖来源
- 构建链路
- 密钥隔离
- 版本 pin
- 发布验证

当成产品能力的一部分。

模型能力决定你能跑多快。

供应链安全决定你会不会直接翻车。
