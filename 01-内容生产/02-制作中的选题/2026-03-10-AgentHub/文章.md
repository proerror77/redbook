# Karpathy 新开源 AgentHub：真正值得看的，是 Agent 怎么协作

现在很多 agent 讨论，本质上还在卷单兵战斗力。

一个 agent 会不会写代码，会不会调工具，会不会自动把活干完。

Karpathy 刚开源的 `agenthub`，我看完后的第一感觉不是“又来一个 agent 项目”，而是：终于有人开始认真想一件更麻烦的事了。

如果将来不是一个 agent，而是 10 个 agent 同时在同一个项目里试验、提交、互相留言，GitHub 这套给人类准备的协作方式，还顺不顺手？

我觉得，这才是这个仓库最值得看的地方。

## 它到底是什么

AgentHub 的官方定位很直接：`GitHub is for humans. AgentHub is for agents.`

说穿了，它就是把协作拆成两件事：

- 代码进一个 `bare git repo`
- 讨论进一个 message board

中间再用一层很薄的 HTTP API 和 CLI 把它们连起来。

架构也很克制：一个 Go 服务、一个 SQLite、磁盘上的 bare repo。

从 README 和代码结构看，push/fetch、提交图谱查询、频道发帖这些基础链路已经搭起来了。

更特别的地方在于，README 明说它没有 `main branch`、没有 `PR`、没有 `merge`，只有一棵往四处长的 commit DAG。

这很反直觉。但也正因为反直觉，才值得看。

## 为什么我觉得它重要

### 1. 它切的不是“Agent 能不能做事”，而是“Agent 怎么一起做事”

现在大多数 agent 产品，优化的还是单兵能力：

- 会不会写
- 会不会调工具
- 会不会自动执行

但真到了复杂任务，瓶颈很快就变了。

问题不再是“一个 agent 能不能干”，而是：

- 谁在试哪条路线
- 哪个 commit 是当前 frontier
- 哪些实验已经失败过
- 新来的 agent 应该从哪一层继续接力

GitHub 当然也能承载这些事，但它的默认假设一直是“人类团队协作”。

分支、PR、merge、review，这些流程对人类很重要。对 agent 未必。

AgentHub 的做法很干脆：先把这套礼仪拿掉，留下一个更适合并行试错的工作台。

### 2. 它把 git 从“版本控制工具”变成了“协作记忆层”

这个项目里最有意思的，不是消息板，而是 commit DAG 被拿来当协作地图。

它直接把 agent 真会关心的图谱查询暴露出来了，比如：

- 某个 commit 的 `children`
- 当前有哪些 `leaves`
- 一条 `lineage` 是怎么长出来的
- 两个 commit 之间到底差了什么

这意味着 agent 不只是把代码 push 上去，还要先看懂这棵实验树现在长什么样，再决定下一步往哪条路继续试。

说白了，git 在这里不只是存代码，更像一层共享工作记忆。

### 3. 它默认多 agent 的世界，本来就不该长得像 GitHub

人类团队需要 PR，是因为要 review、要讨论、要签字、要控制风险。

但 agent 团队未必一样。

很多时候，agent 更像研究员或者实习生。先大量尝试，再回头筛结果。

如果一开始就把人类工作流原封不动套到 agent 身上，协作成本会很高。每个 agent 都在等流程，不是在跑实验。

我自己的解读是，它更在意先把并行探索跑起来，再慢慢讨论怎么收敛。

这不是 README 的原话，是我看完这个设计后的判断。

对做 research agent、code swarm、自动化实验平台的人来说，这个判断很有启发。

## 但别急着把它吹成“下一个 GitHub”

这里也得泼点冷水。

AgentHub 现在还是非常早期的探索项目。README 里直接写了：`Work in progress. Just a sketch. Thinking...`

GitHub API 显示，仓库创建时间是 `2026-03-09T19:21:58Z`，换算到北京时间是 `2026-03-10 03:21:58`。也就是说，这真的是一个刚开出来的仓库。

所以这东西现阶段更适合这样理解：

- 它是一个方向样本，不是成熟平台
- 它适合研究型、多 agent 试验型场景
- 它还不是大多数团队今天就能拿来替代 GitHub 的东西

不过，早期不等于不重要。

恰恰因为它早，你更容易看出这类项目在试什么方向。

它关心的不是“怎么让一个 agent 更像程序员”。

它更像是在追问：如果未来真的有一群 agent 在网上一起干活，底层协作层该长什么样？

这个问题，比再做一个 agent demo 大得多。

## 对一人团队，这事为什么也值得看

很多人会觉得，多 agent 协作离自己还远。

其实不远。

今天很多一人团队的工作方式，已经慢慢变成“1 个人 + 1 个 AI”。

我的判断是，再往后很可能会变成“1 个人 + 多个 agent”。

到那个阶段，难点就不是 prompt 了。

难点会变成：

- 怎么分工
- 怎么接力
- 怎么避免重复试错
- 怎么知道哪个结果值得保留

所以 AgentHub 值得看的，不只是这个仓库能不能跑起来。

更重要的是，它提前把一个更难的问题摆到了台面上：

当 agent 开始成群结队工作，协作层应该怎么设计？

## 我最后的判断

如果你只是想找一个现成工具，今天就拿来搭自动化团队，那 AgentHub 还不是答案。

但如果你在做 agent infra、research automation，或者你已经感觉到“单个 agent 能干活”不再是最难的部分了，那这个仓库值得认真看一遍。

因为它戳中的不是能力层，而是组织层。

我的押注是，下一个阶段真正拉开差距的，未必是谁家的 agent 更会写代码。

而是谁先把“多 agent 怎么协作、怎么试错、怎么留痕、怎么接力”这层基础设施搭起来。

AgentHub 还很粗。

但方向是对的。
