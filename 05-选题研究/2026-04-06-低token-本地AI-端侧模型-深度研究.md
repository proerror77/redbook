# 低 token / 本地 AI / 端侧模型 - 深度研究

> 研究日期：2026-04-06
> 研究目标：解释为什么 AI 正从“拼能力”转向“拼效率 / 成本 / 本地化”
> 关键词：Caveman、Gemma 4、Claude Code、Ollama、LM Studio、AI Edge Gallery

## 一句话结论

AI 的竞争结构开始变了。

过去大家主要比“谁的模型更强”。
现在越来越多场景开始比：

- 谁能少用 token
- 谁能把任务放到本地跑
- 谁能把 agent 直接放到设备端跑

也就是说，AI 正在从单纯的“能力竞赛”，转向“能力 + 成本 + 延迟 + 隐私 + 部署位置”的综合竞赛。

## 这不是一个趋势，是三层变化同时发生

### 1. 低 token：先把废话砍掉

这层不是换模型，而是让同一个模型少说废话、少吐 token。

代表案例是 `Caveman`。

- `JuliusBrussee/caveman` 的 README 直接把它定义成一个让 Claude Code / Codex “像穴居人一样说话”的 skill/plugin。
- 仓库给出的 benchmark 显示：平均从 `1214` 个 token 降到 `294` 个 token，平均节省 `65%`，README headline 直接写的是“cuts ~75% of tokens”。
- 它特别强调：**只影响 output tokens，不影响 reasoning tokens**。原话意思很明确，不是脑子变小了，是嘴变小了。

这件事为什么重要？

- 因为 agent 工作流天然比单轮问答更容易“话多”。
- 一旦进入多轮对话、工具调用、代码解释、日志说明，输出 token 很快就会膨胀。
- 当 output token 是单独计费项时，少说废话就不只是文风问题，而是速度和成本问题。

结论：

- `低 token` 不是小技巧，而是 AI 进入生产阶段后的工程优化。
- 以前大家忍受冗长回答，是因为 novelty 更重要。
- 现在真正高频使用的人，已经开始嫌它“太啰嗦、太贵、太慢”。

### 2. 本地 AI：把推理从云 API 拉回自己的机器

第二层变化不是“少说”，而是“少走云”。

本地 AI 的核心不是情怀，而是三个现实收益：

- 更低的边际成本
- 更可控的延迟
- 更强的数据控制权

这层最关键的信号，不是某个开源模型本身，而是**本地模型终于能接入成熟 agent 工具链**。

#### Ollama 这一侧

Ollama 官方文档已经把 `Claude Code` 作为正式 integration 写进去了。

- 文档明确写：open models can be used with Claude Code through Ollama's `Anthropic-compatible API`
- 提供了 `ollama launch claude` 的 quick setup
- 也支持 headless mode
- 但同时明确提醒：`Claude Code requires a large context window`，推荐至少 `64k tokens`

这说明两件事：

- 第一，本地模型不再只是“聊天玩具”，而是开始接 coding agent 这种重工作流。
- 第二，本地替代不是零门槛的。真正想跑 coding/agent workload，context window 和硬件资源仍然是硬约束。

Ollama 的 Anthropic compatibility 文档还给了更现实的提醒：

- 推荐的 coding 模型里，`qwen3-coder` 是 `30B` 量级
- 文档明确写它需要至少 `24GB VRAM` 才能顺畅运行

所以“本地 AI”不是免费午餐。
它是在把 API 成本，换成你自己的显存、机器和部署复杂度。

#### LM Studio 这一侧

LM Studio 官方开发文档的信号更强。

- 文档明确写它提供本地 `REST API`、`OpenAI-compatible` endpoint、`Anthropic-compatible` endpoint
- 明确写可以做 `tool calling and local agents with MCP`
- 还把 `llmster` 定义成可以跑在 server / cloud instance / CI 上的 headless daemon

这意味着什么？

- 本地模型不再只能在 GUI 聊天框里玩
- 它开始具备“像基础设施一样被接入”的条件
- 也就是：可以被 IDE、agent、脚本、CI、自动化系统稳定调用

结论：

- 本地 AI 真正的转折点，不是开源模型变强一点
- 而是 **本地推理终于开始长出“工程接口”**

### 3. 端侧模型：不是把云模型缩小，而是把 agent 放到设备上

第三层比“本地 AI”再进一步。

本地 AI 可以跑在你的笔记本、工作站、家里服务器。
端侧模型则是直接跑在手机、浏览器、IoT、终端设备上。

这层的核心目标不是“最高智商”，而是：

- 离用户更近
- 离传感器更近
- 离操作界面更近
- 离隐私数据更近

## Gemma 4 为什么是这个趋势的关键证据

Google 在 2026-04-02 发布的官方材料非常直白：

- Google Developers Blog 的标题就叫：`Bring state-of-the-art agentic skills to the edge with Gemma 4`
- 文中明确写：Gemma 4 可以让开发者“go beyond chatbots”去构建 **agents and autonomous AI use cases running directly on-device**
- 还点名能力包括：`multi-step planning`、`autonomous action`、`offline code generation`

这不是“本地聊天”。
这是 Google 在官方口径里直接把“agentic skills”与“edge”绑定。

Gemma 4 model card 也给了几个非常关键的硬指标：

- 支持最长 `256K` context
- 模型尺寸覆盖 `E2B`、`E4B`、`26B A4B`、`31B`
- 官方明确写它们可部署在 `high-end phones`、`laptops`、`servers`
- 小模型被明确描述为 `optimized for on-device`
- 官方直接写有 `native function-calling support`
- 官方直接写 `enhanced coding & agentic capabilities`

这说明什么？

- Google 不是把端侧 AI 当 demo
- 而是在认真做一条从手机到桌面到 IoT 的统一部署路线

## AI Edge Gallery 不是 demo app，而是产品方向样板

`Google AI Edge Gallery` 很值得重视，因为它不是论文，不是 benchmark，而是面向真实设备体验的产品样板。

截至 `2026-04-06`，App Store 页面已经写得很清楚：

- `Gemma 4 is here`
- `Run the latest high-performance models fully offline`
- `Agent Skills`
- `Thinking Mode in AI Chat`
- 支持社区自定义 skill loading
- 强调 `100% On-Device Privacy`

Google 官方博客对它的定位也很明确：

- 它允许开发者 build and experiment with AI experiences that run entirely on-device
- `Agent Skills` 被定义为最早一批可以 **entirely on-device** 运行 multi-step autonomous workflow 的应用

Google 在这篇博客里甚至给了很具体的能力方向：

- 用 skill 查询 Wikipedia 做知识增强
- 把段落或视频转成 summary / flashcards / graphs
- 接入其他模型做 TTS、图像生成、音乐生成
- 用对话直接串起完整 end-to-end workflow

这已经很像“新 App 层”了。

不是“手机里跑一个小模型”。
而是“手机里长出一个可调用 skill 的 agent runtime”。

## LiteRT-LM 说明了端侧路线已经开始工程化

如果说 AI Edge Gallery 负责展示产品形态，那 `LiteRT-LM` 负责证明这条路线正在被工程化。

Google 官方博客给出了几个特别关键的指标：

- 在部分设备上，Gemma 4 E2B 可在 `<1.5GB` 内存下运行
- 支持 constrained decoding，适合 tool calling 和结构化输出
- 支持 dynamic context，能利用 Gemma 4 的 `128K` context
- 在 agentic use case 下，可在 `2` 个 distinct skills、`4000` input tokens 的情况下，`under 3 seconds` 完成处理
- CLI 已支持 tool calling，这正是 AI Edge Gallery 里 Agent Skills 背后的能力

这说明：

- 端侧 agent 不是只在做“跑起来”
- 已经在开始优化 memory、context、tool use、runtime performance 这些真正决定能否落地的指标

## 为什么现在这条线成立

### 1. 成本开始反噬冗长工作流

Anthropic 和 OpenAI 的官方 pricing 文档都把 `output tokens` 单列出来收费。

这意味着：

- 模型不只是“答得对不对”
- 还要看“答这一次花了多少 token”

在高频 agent 场景里，这个问题会被放大：

- 多轮上下文更长
- 工具调用会插入更多结构化上下文
- 解释型输出往往很冗长

所以 `Caveman` 这种“把嘴变小”的东西会火，不是偶然。

### 2. 本地基础设施成熟了

以前“本地模型”最大的问题不是模型不够强，而是：

- 没有稳定 API
- 没有 agent 兼容层
- 很难接入现有工具

现在：

- Ollama 提供 Anthropic-compatible API，可以接 Claude Code
- LM Studio 同时提供 OpenAI-compatible / Anthropic-compatible endpoint
- LM Studio 还支持 local agents with MCP 和 headless daemon

也就是说，本地模型第一次开始像“后端服务”而不是“个人玩具”。

### 3. 端侧路线有了官方产品和 runtime

以前讲 on-device AI，更多是概念演示。
现在 Google 已经把这条线补齐成一套组合拳：

- Gemma 4 模型族
- AI Edge Gallery 产品样板
- LiteRT-LM runtime
- Android AICore / ML Kit GenAI API

这说明大厂已经不把端侧当彩蛋，而是当一条正式路线。

## 但这条线也有明显边界

### 1. 低 token 不是万能药

- 它主要砍的是 output 冗余
- 不会自动降低 reasoning 开销
- 对某些需要解释、教学、安抚用户的场景，过度压缩会损失体验

### 2. 本地 AI 不是“零成本”

- 它只是把成本从 API 账单换成了显卡、内存、维护复杂度
- 真正能跑 coding agent 的模型，对 context 和显存要求仍然高

### 3. 端侧模型不是要替代云

Android 官方文档写得很清楚：

- 复杂任务、高能力需求、跨平台一致性，云模型仍有优势
- 端侧更适合离线、低延迟、隐私敏感、成本敏感场景

换句话说，未来不是“全上云”或“全下端”。
而是**任务分层**。

## 最值得写的核心观点

### 观点 1

AI 下一阶段不再只是比谁最聪明，而是比谁最省、最快、最靠近用户。

### 观点 2

`token` 正在变成新的“响应时间 + 毛利率”指标。

以前大家看 model benchmark。
以后真正做产品的人，会越来越在意：

- 一次回答多少 token
- 一次 agent loop 吃掉多少上下文
- 这件事到底值不值得走云

### 观点 3

本地 AI 的真正价值不是“离线聊天”，而是把高频、低风险、对隐私敏感的工作从云端拿回来。

### 观点 4

端侧模型的终局不是“小号 ChatGPT”，而是“设备上的 agent runtime”。

### 观点 5

真正的产品分层会变成：

- 云端：最复杂、最开放、最需要高能力的任务
- 本地：高频、可控、可缓存、希望降本的任务
- 端侧：即时、隐私敏感、依赖设备上下文的任务

## 最终判断

这条趋势的本质不是“模型更小了”。

而是：

- AI 已经从炫技阶段，进入部署阶段
- 从“能不能做”转向“该在哪做、花多少钱做、是否值得做”

所以这波最重要的变化，不是某个模型分数涨了几点。
而是 AI 系统开始出现新的优化顺序：

**先压 token，再拉本地，最后推到端上。**

这才是 `Caveman`、`Gemma 4`、`Claude Code + Ollama/LM Studio` 放在一起时真正共同指向的事。
